# 기술 부채 추적기 (Tech Debt Tracker)

## 이 문서의 목적
"나중에 고쳐야 하는데 지금은 시간이 없어서 넘어간" 것들을 기록한다.
Phase별로 의도적으로 남긴 기술 부채와 향후 해결 계획을 추적한다.

---

## 현재 알려진 기술 부채

### TD-001: DB 없이 파일시스템 기반
- **발생 시점**: 설계 단계 (core-beliefs.md — 단순함 우선)
- **내용**: 영상, 세그멘테이션, 메타데이터를 모두 파일시스템에 저장
- **위험**: 동시 접속자가 많아지면 파일 잠금, 검색 성능 문제
- **해결 시점**: 동시 접속자 10명 이상 또는 저장 파일 1,000개 이상일 때
- **해결 방안**: SQLite → PostgreSQL 순으로 점진적 도입
- **우선순위**: 낮음 (현재 5~15명 규모에서는 문제 없음)

### TD-002: 인증/인가 없음
- **발생 시점**: 설계 단계 (내부망 전용)
- **내용**: 누구나 접속하여 모든 기능을 사용할 수 있음
- **위험**: 외부 네트워크에 노출되면 보안 문제
- **해결 시점**: 외부 접근이 필요해질 때
- **해결 방안**: FastAPI OAuth2 + JWT 또는 역방향 프록시 인증
- **우선순위**: 낮음 (내부망에서만 사용하는 한)

### TD-003: 단일 GPU 전제 설계
- **발생 시점**: 설계 단계 (gpu-memory-management.md)
- **내용**: InferenceService가 단일 GPU(cuda:0)만 사용
- **위험**: GPU 추가 시 설계 변경 필요
- **해결 시점**: GPU 2대 이상이 될 때
- **해결 방안**: device 인덱스 파라미터화 → 분산 inference 서비스 분리
- **우선순위**: 낮음 (대부분 단일 GPU 환경)

### TD-004: Inference 큐가 인메모리
- **발생 시점**: Phase 1 (asyncio.Queue 사용)
- **내용**: 서버 재시작 시 큐의 대기 작업이 사라짐
- **위험**: inference 중 서버 크래시 시 작업 유실
- **해결 시점**: 안정성 요구가 높아질 때
- **해결 방안**: Celery + Redis로 전환
- **우선순위**: 중간 (크래시 시 다시 실행하면 되므로)

### TD-005: Frontend 에러 핸들링 최소화
- **발생 시점**: Phase 1 (MVP 우선)
- **내용**: 네트워크 오류, 타임아웃 등의 예외 처리가 최소한
- **위험**: 에러 발생 시 사용자가 흰 화면만 보게 됨
- **해결 시점**: Phase 2 이후
- **해결 방안**: React Error Boundary + 전역 에러 핸들러
- **우선순위**: 중간

### TD-006: 테스트 코드 부재
- **발생 시점**: Phase 0~1 (빠른 구현 우선)
- **내용**: 자동화된 테스트(unit, integration)가 없음
- **위험**: 코드 수정 시 기존 기능이 깨지는지 확인 불가
- **해결 시점**: Phase 2 완료 후
- **해결 방안**: pytest (Backend) + React Testing Library (Frontend)
- **우선순위**: 중간 (기능이 안정화된 후)

## 기술 부채 추가 규칙

새로운 기술 부채를 발견하면 아래 형식으로 추가한다:

```markdown
### TD-XXX: [제목]
- **발생 시점**: [언제, 어떤 결정에서 발생했는가]
- **내용**: [무엇이 문제인가]
- **위험**: [방치하면 어떤 일이 생기는가]
- **해결 시점**: [어떤 조건이 되면 해결해야 하는가]
- **해결 방안**: [어떻게 해결할 것인가]
- **우선순위**: 높음 / 중간 / 낮음
```
