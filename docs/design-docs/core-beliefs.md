# Core Beliefs — 프로젝트 설계 철학

## 이 문서는 무엇인가
모든 설계 결정의 기반이 되는 "믿음"들을 정리한 문서이다.
구현 중에 여러 선택지가 있을 때, 여기 적힌 원칙에 더 부합하는 쪽을 선택한다.

---

## 믿음 1: 연구자의 시간이 가장 비싸다

### 배경
우리가 이 뷰어를 만드는 이유는 단순하다.
nnUNet inference를 터미널에서 돌리고, 결과를 ITK-SNAP에서 열어보고,
수정이 필요하면 다시 스크립트를 짜는 — 이 반복적인 과정이 너무 오래 걸리기 때문이다.

연구원 한 명이 하루에 확인해야 할 케이스가 10~20개라면,
매 케이스마다 터미널 명령어 입력 → 파일 경로 복사 → ITK-SNAP 실행 → 결과 비교를
반복하는 것은 연구 시간의 낭비다.

### 이 원칙이 만드는 결정
- **브라우저만 열면 바로 사용 가능**해야 한다 → 복잡한 설치 과정 금지
- **업로드 → 결과 확인까지 클릭 3번 이내**를 목표로 한다
- **설정이 필요 없는 합리적 기본값**을 제공한다 (예: CT Abdomen 윈도우/레벨 기본 적용)
- **에러 메시지는 해결 방법까지 안내**한다 (예: "GPU 메모리 부족 → fold 수를 줄여보세요")

## 믿음 2: 의료영상 데이터는 한 비트도 손상되면 안 된다

### 배경
의료영상 분석에서 가장 위험한 버그는 "결과가 나오긴 하는데 살짝 틀린" 경우다.
NIfTI 헤더의 affine matrix가 잘못되면 좌우가 뒤집히고,
spacing이 틀리면 장기 크기가 달라지며,
HU 값이 정규화되면 측정값이 의미를 잃는다.

이런 버그는 눈으로 봐서는 발견하기 어렵고,
이를 기반으로 내린 연구 결론이 잘못될 수 있다.

### 이 원칙이 만드는 결정
- **NIfTI 헤더(affine, spacing, qform/sform)를 어떤 처리에서도 보존**한다
- **CT HU 값은 저장 단계에서 절대 변환하지 않는다** (윈도우/레벨은 표시만)
- **세그멘테이션 저장 시 원본 영상의 공간 정보를 그대로 복사**한다
- **RAS+ 변환은 명시적으로 수행**하고, 원본 파일은 항상 보존한다
- **편집 저장 전 shape 일치를 반드시 검증**한다

---

## 믿음 3: nnUNet은 블랙박스가 아니라 존중해야 할 파이프라인이다

### 배경
nnUNet의 강점은 단순히 네트워크 구조가 아니라, 전체 파이프라인에 있다.
자동 전처리(resampling, normalization), 학습 전략(5-fold CV),
후처리(connected component analysis), 앙상블까지 — 이 모든 것이 유기적으로 설계되어 있다.

과거에 "nnUNet 결과를 개선해보겠다"고 전처리나 후처리를 임의로 수정했다가
오히려 성능이 떨어진 경험이 있다. nnUNet의 파이프라인은 그 자체로 최적화되어 있으므로,
우리가 건드릴 부분은 최소화해야 한다.

### 이 원칙이 만드는 결정
- **nnUNet의 공식 Python API(nnUNetPredictor)를 그대로 사용**한다
- **전처리/후처리를 자체 구현하지 않는다** (nnUNet 내부에서 자동 수행)
- **dataset.json을 레이블 정보의 유일한 출처**로 사용한다
- **nnUNet_results 폴더 구조를 수정하지 않는다** (읽기 전용 취급)
- **nnUNet 업데이트 시 우리 코드를 최소한만 수정**하면 되도록 얇은 래퍼만 작성한다

## 믿음 4: 단순한 시스템이 유지보수 가능한 시스템이다

### 배경
이 프로젝트의 주 개발자와 유지보수자는 웹 개발 전문가가 아닌 연구자들이다.
복잡한 아키텍처(마이크로서비스, 메시지 큐, ORM, 복잡한 상태관리)는
처음에는 "확장성을 위해" 도입하지만,
결국 아무도 이해하지 못하는 코드가 되어 방치되는 경우가 많다.

### 이 원칙이 만드는 결정
- **모놀리식 Backend** (마이크로서비스 분리 금지)
- **DB 없이 파일시스템 기반** (PostgreSQL, MongoDB 등 도입 금지)
- **인증/인가 없음** (내부망 전용이므로)
- **최소한의 추상화** (불필요한 인터페이스/팩토리/DI 패턴 금지)
- **환경변수 + .env 파일로 설정 관리** (별도 설정 서버 금지)
- **Docker Compose 한 줄로 실행** 가능해야 한다

---

## 믿음 5: 완벽한 것보다 동작하는 것이 먼저다

### 배경
연구 도구는 논문 마감이나 학회 발표 등 시간 압박이 항상 있다.
"완벽한 3D 편집 도구"를 6개월 동안 만드는 것보다,
"2D 슬라이스 뷰어 + 기본 inference"를 2주 안에 만들어서 당장 쓸 수 있게 하는 것이 낫다.

### 이 원칙이 만드는 결정
- **P0 기능만으로 사용 가능한 MVP를 먼저 완성**한다
- **P1, P2 기능은 MVP가 동작한 후에 점진적으로 추가**한다
- **과도한 최적화보다 정확한 동작을 우선**한다
- **엣지 케이스 처리보다 주요 시나리오의 안정성을 우선**한다

---

## 원칙 간 충돌이 생기면?

우선순위:
1. **데이터 무결성** (믿음 2) — 이건 절대 양보하지 않는다
2. **nnUNet 파이프라인 존중** (믿음 3) — inference 정확성 보장
3. **연구자의 시간** (믿음 1) — 사용성
4. **단순함** (믿음 4) — 유지보수성
5. **동작하는 것 먼저** (믿음 5) — 속도

예를 들어, "사용자 편의를 위해 HU 값을 자동 정규화해서 저장하자"는 제안이 있으면
→ 믿음 2(데이터 무결성) > 믿음 1(편의성) 이므로 거부한다.

"코드가 좀 복잡해지더라도 nnUNet의 원래 전처리를 유지하자"는 제안이 있으면
→ 믿음 3(nnUNet 존중) > 믿음 4(단순함) 이므로 수용한다.
